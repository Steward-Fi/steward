# Privy Parity Audit

Date: 2026-06-05

This audit compares Steward against Privy's public documentation and OpenAPI
surface. Sources reviewed:

- Privy docs welcome and agent index: https://docs.privy.io/welcome,
  https://docs.privy.io/llms.txt, https://docs.privy.io/llms-full.txt
- Privy REST OpenAPI: https://api.privy.io/v1/openapi.json
- Focused 2026-05-31 and 2026-06-03 refresh pages: guest accounts, automatic wallet
  creation, chain support, accounts, wallet create, HD wallets, new-device
  recovery, EIP-7702 authorization, Spark BTC/Lightning RPC, and external
  wallet connector configuration.
- 2026-06-03 docs index/API refresh: account balance/aggregation resources,
  authorization keys, owners, signers, nested key quorums, manual-approval
  intent types, wallet external IDs, connect-or-create wallet UX, Spark wallet
  actions, swap/earn action APIs, SDK language coverage, and request hardening
  docs.
- 2026-06-04 local surface refresh: `agent_signers` P-256 public-key support,
  nested key-quorum schema/middleware tests, account aggregation aliases,
  validation matrix, SDK route helpers, dashboard Account/Settings surfaces,
  current API/web route names, React Native auth bootstrap, app-client embedded
  wallet creation overrides, platform wallet-link policy enforcement, and SPL
  Token/Token-2022 transfer actions.
- 2026-06-04 OpenAPI hardening refresh: generated API docs now inventory
  sensitive `/auth` MFA/session mutations with request-expiry,
  authorization-signature, idempotency, one-time secret/token, and recent-MFA
  requirements.
- 2026-06-05 encrypted import refresh: vault agent and authenticated user-wallet
  imports now have one-time X25519/HKDF/AES-GCM import sessions, SDK helpers,
  generated OpenAPI coverage, replay/plaintext/mismatch rejection tests,
  corrected API reference docs, and a first-pass hosted React user-wallet
  import component.
- 2026-06-05 automated OpenAPI parity refresh: `scripts/privy-openapi-parity.ts`
  compares Privy's live OpenAPI against Steward's generated contract. After
  adding Privy-shaped fiat aliases, the live classifier sees coverage across all
  12 route families it tracks; product-depth gaps remain in provider-backed
  money movement surfaces.
- Steward codebase and docs in this repository.

Legend:

- `Covered`: Steward has an implemented local equivalent.
- `Partial`: Steward has primitives or adjacent features, but not product/API parity.
- `Gap`: Open-source feasible feature missing or not wired.
- `Adapter`: Steward can expose an open adapter, but production parity depends on
  regulated, commercial, liquidity, chain-indexing, or compliance providers.
- `Not OSS-only`: Cannot be fully satisfied by code alone, though interfaces can
  be open source.

## Executive Summary

Steward already covers a meaningful Privy-like core for agent and application
wallet infrastructure: passkeys, email magic links, SIWE/SIWS, OAuth for several
providers, JWT refresh/revocation, multi-tenant API keys, encrypted EVM and
Solana wallets, signing, import/export, approval queues, policies, webhooks,
React components, SDK, dashboard, proxy credential injection, audit trails, and
self-hosted deployment.

The largest open-source feasible gaps are:

1. Dashboard UX for richer end-user account-linking flows. A first-pass
   authenticated Account page now lists primary login methods, linked accounts,
   and embedded wallets, supports self-unlink through the SDK, persists
   dashboard sessions across magic-link redirects with session-scoped storage,
   and has Chromium e2e screenshot coverage. `@stwd/react` also exports a
   reusable linked-account review/unlink component with SMS/WhatsApp OTP,
   EVM/Solana signed-wallet, OAuth code-exchange, Telegram, and Farcaster
   linking controls. Remaining work is deeper hosted redirect/widget
   orchestration and dashboard parity for those provider-specific flows.
2. SMS/WhatsApp auth routes, TOTP/passkey/SMS MFA routes, and wallet-action MFA
   enforcement now cover the core OSS-sensitive action paths. Generated OpenAPI
   now documents hardening headers and one-time MFA/session token behavior for
   sensitive `/auth` MFA/session mutations. Remaining work is richer hosted
   recovery, provider setup, and dashboard polish.
3. Richer claim-mapping UI and operational docs for tenant-scoped bring-your-own
   JWT/OIDC providers.
4. Privy-style authorization keys, owners, signers, key quorums, and generic
   intent lifecycle for manual approval of wallet, policy, and quorum changes.
5. Richer policy conditions for typed data, EIP-7702, Solana instructions,
   time, and stateful aggregations. Steward now has basic reusable condition
   sets for string membership checks and a hard EVM contract/function selector
   allowlist for calldata-bearing transactions.
6. Wallet action APIs for swap quotes/swaps, earn vault operations, richer
   transfer settlement, and consistent action webhooks. Steward now has basic
   native EVM transfer quote/create/status routes over the existing vault.
7. Account aggregation APIs over user and agent wallets, multiple chains, and
   richer custody/portfolio metadata. Agent and authenticated-user account
   aggregation now include best-effort native/ERC20 portfolio assets and USD
   valuation.
8. Gas sponsorship and relayer/paymaster abstractions.
9. Hardware/HSM-backed import and custody flows. Steward now has encrypted
   vault and authenticated user-wallet import sessions, SDK helpers,
   `@stwd/react` hosted user-wallet import UX, Account dashboard entry-point
   coverage, Redis/Postgres import-session readiness checks, tenant hard
   import/export disables, and per-surface MFA freshness. Remaining parity is
   external signer/HSM-backed enterprise import flows and custody isolation
   beyond the API process.
10. Transaction indexing and richer webhook coverage for confirmed, failed,
   replaced, pending, provider-error, deposit, and withdrawal events.
11. Broader SDK/platform coverage: React Native, iOS, Android, Flutter, Unity,
    Go, Java, Rust, Ruby, and Python. React Native, Swift/iOS, and first-pass
    SDKs for Go, Java, Rust, Ruby, Python, and C#/.NET now exist.
12. Dashboard controls for SSO, team RBAC, and organization/app secrets.
    Login-method, CAPTCHA, allowlist/denylist, app-origin/redirect, app-client,
    appearance, and OIDC provider controls now have first-pass
    dashboard/API/SDK coverage.
13. Security hardening for short-lived access tokens, bot controls, security
    checklist, and threat-model alignment. Idempotency keys, request expiry, and
    opt-in authorization signatures are now implemented for sensitive API calls.
14. Latest Privy docs refresh adds explicit gaps for Spark BTC/Lightning product
    APIs, broader Tier-2/Tier-1 chain wallet abstractions, guest-account
    lifecycle UX, automatic wallet creation on login, account CRUD and
    aggregation over wallet groups, wallet external IDs, connect-or-create
    wallet UX, richer external-wallet connector configuration, nested key
    quorums, and typed manual-approval intents for policy, wallet, rule, and
    quorum changes.
15. Local 2026-06-04 refresh shows Steward has advanced the authorization-key
    lane: agent signer routes, SDK helpers, API docs, and generated OpenAPI now
    expose HMAC or P-256 public-key signers, while key-quorum routes expose
    nested child quorum IDs. Nested key-quorum verification is tested.
    Generic non-vault intent review now re-checks live owner/admin tenant
    membership so removed or deactivated reviewers cannot approve, reject,
    cancel, execute, or fail stale-token requests. Remaining Privy parity is
    resource-level authorization keys beyond agent signers, owner/signer
    assignment across user wallets and policies, and route coverage beyond
    agent vault actions.

Provider/compliance dependent areas should be adapter-first:

- SMS/WhatsApp deliverability, fiat on/off-ramp, KYC/TOS, ACH/wire/SEPA, card
  funding, Kraken/exchange embed, licensed custody, curated yield distribution,
  SOC2 certification, and Privy's exact managed TEE/key-splitting network.

## Recommended Implementation Waves

### Current Highest-Impact Backlog

This ordering excludes account portfolio, generic intents, contract allowlists,
selector-gated ERC20 transfer actions, and conservative SPL Token/Token-2022
transfer actions that already have first-pass implementation. Settled token
accounting and ATA creation UX remain open.

1. End-user linked account management: authenticated users can list and
   self-unlink owned linked accounts with last-method-unlink,
   session-revocation, refresh-token cleanup, and tenant/user isolation tests.
   Ethereum and Solana wallet self-link now have one-time signed proof, replay,
   duplicate, and cross-user ownership tests. OAuth self-link uses authenticated
   provider code exchange with encrypted token storage and ownership tests. Add
   provider-specific proof/link flows for phone/WhatsApp OTP and
   Telegram/Farcaster one-time social proofs are implemented. Passkeys are
   surfaced as linked accounts and can be self-unlinked with last-method
   protection. The dashboard Account page now lists primary methods, linked
   accounts, and embedded wallets, supports unlink, and is covered by Chromium
   e2e screenshot verification. `@stwd/react` exposes the same first-pass
   review/unlink surface as `StewardLinkedAccounts`, plus SMS/WhatsApp OTP,
   EVM/Solana signed-wallet, OAuth code-exchange, Telegram, and Farcaster link
   controls. Add richer hosted OAuth/social dashboard orchestration.
2. Dashboard/app auth configuration parity: login method toggles, CAPTCHA,
   email/wallet/phone policy controls, app-origin/redirect controls, and tenant
   OIDC provider controls now have API/SDK/dashboard coverage. App clients and
   whitelabel theme controls now have dashboard preview plus browser
   verification. Team RBAC now has API/SDK/dashboard role editing with MFA,
   owner-lifecycle guards, audit events, and browser verification. Enterprise
   OIDC and SAML SSO now cover auth-code redirects, verified-domain
   enforcement, SAML group-to-role mapping, signed XML fixtures, and the
   latest red-team fixes for direct-JWT, RelayState, email-attribute, and
   passkey SSO-required-domain bypasses. Pending invite rows, one-time invite
   tokens, verified-email acceptance, optional email delivery through tenant
   email providers, SDK helpers, dashboard create/revoke UI, Chromium e2e, and
   PGLite constraints now exist. Remaining work is deeper
   OAuth/IdP setup docs, richer invite branding, logo/assets, and
   hosted-widget preview polish.
3. Gas sponsorship and bundler/paymaster submission: tenant sponsorship
   config, mock reservations, sponsored-gas ledger persistence, rolling
   wallet/tenant cap checks, client-sponsorship gating, direct and manually
   approved transfer reservations under tenant advisory locks, and a
   Privy-style platform gas-spend query now exist. Remaining work is real
