# ADR-0001 — Cryptographic choices for vault and secret storage

**Status:** Accepted
**Last reviewed:** 2026-04-24

## Context

Steward holds agent signing keys (EVM and Solana) and arbitrary tenant secrets
(`packages/vault/src/secret-vault.ts`). The threat we are explicitly designing
for is: **an attacker with read access to the Postgres row data must not be
able to trivially recover plaintext keys, and must be forced to spend
meaningful CPU to attempt a password-guessing attack if they also obtain the
master password hash.**

We are **not** designing to defend against an attacker with code execution on
the API process — at that point the master password is in process memory and
the keys are decryptable by construction. That bound is documented in
`docs/security/threat-model.mdx`.

This ADR records the primitives and parameters actually used in
`packages/vault/src/keystore.ts` and explains the choices.

## Decision

### Symmetric encryption: AES-256-GCM

Implemented in `keystore.ts` via `createCipheriv("aes-256-gcm", derivedKey, iv)`
(line ~59) and `createDecipheriv("aes-256-gcm", …)` (line ~79).

**Why AES-256-GCM:**

- AEAD (confidentiality + integrity in one primitive). We use the auth tag
  (`cipher.getAuthTag()`, stored as `tag`) to detect tampering of ciphertext,
  IV, or salt at decrypt time — a bit-flipped row fails `final()` rather than
  silently decrypting to garbage.
- Shipped in Node's `crypto` module and every modern runtime (no extra deps).
- Hardware-accelerated on any x86_64 with AES-NI and every ARMv8 with the
  crypto extension. In a Docker deployment the API container inherits the host
  acceleration for free.

**Why not ChaCha20-Poly1305:** ChaCha is a better fit when AES-NI is
unavailable (very low-end ARM, constant-time constraints on secret-dependent
table lookups in software AES). Our deployment target is Linux x86_64 /
ARMv8 server hardware, so this advantage does not apply. Node also ships
`chacha20-poly1305` via `crypto.createCipheriv`, so swapping is a one-line
change if a future deployment target justifies it.

**Why not libsodium sealed boxes / `crypto_secretbox`:** `crypto_secretbox` is
XSalsa20-Poly1305 — same hardware-acceleration argument as ChaCha. Sealed
boxes (libsodium's `crypto_box_seal`) are public-key crypto intended for
"anyone can encrypt, only the holder of the private key can decrypt," which
is not the vault's model. The vault is purely symmetric: the server
re-encrypts and decrypts with a key it already has. Introducing libsodium
would add a native-module dependency (Bun's `sodium-native`) for no functional
gain.

### IV/nonce

16 random bytes per encrypt, generated via `randomBytes(16)` in
`keystore.ts` line ~57. GCM's standard nonce length is 12 bytes; Node accepts
16 and this is fine because:

- We generate the nonce fresh per record with a CSPRNG, so the birthday bound
  is 2^64 encryptions under the same key before collision becomes probable
  (well beyond any deployment scale we plan for).
- Nonce and tag are stored alongside ciphertext in the row, so recovery on
  decrypt is deterministic.

We will **not** switch to a deterministic nonce (e.g. a counter) because the
vault does not own a monotonic counter it can safely coordinate across
replicas.

### KDF: scrypt

Implemented via `scryptSync(masterPassword, salt, 32)` in `keystore.ts`
line ~48 (root key derivation) and again per-record in `encrypt()` /
`decrypt()` (lines ~61, ~82).

**Parameters:** We rely on Node's defaults:
- `N = 2^14 = 16384` (cost)
- `r = 8` (block size)
- `p = 1` (parallelization)
- `maxmem` default

These are the Node 20+ defaults. This is **weaker** than current OWASP
guidance for password storage, but stronger than our actual requirement — we
are not storing user-facing password hashes here, we are deriving an
encryption key from a server-side master secret that is itself generated by
the operator. The operator SHOULD treat `STEWARD_MASTER_PASSWORD` as a
high-entropy secret (≥256 bits), not as a human-memorable password. In that
regime, scrypt's memory-hardness is defense-in-depth, not the primary
protection.

**Why scrypt (vs PBKDF2, Argon2):**

- **vs PBKDF2:** scrypt is memory-hard; PBKDF2 is only CPU-hard. Against an
  attacker with a large GPU farm, PBKDF2 at realistic iteration counts (e.g.
  600k) is cheaper to brute-force than scrypt at N=2^14. PBKDF2 is still
  FIPS-blessed, which may matter for some enterprise deployments; we'll
  revisit if a pluggable KDF backend is requested.
- **vs Argon2:** Argon2id is the modern favorite and probably what we would
  pick for a greenfield design today. We chose scrypt because it's in Node's
  stdlib (`node:crypto`) and requires no native module or `Bun.password`
  shim. The gain from Argon2id for our use case (deriving a key from a
  high-entropy master secret) is marginal. We'll re-evaluate if/when we move
  to a pluggable KDF.

### Per-record random salt

`encrypt()` generates `randomBytes(16)` as the per-record salt
(`keystore.ts` line ~58) and stores it alongside the ciphertext. The
per-record salt is used as the input to a second `scryptSync` call that
derives the actual AES key for that record from the already-derived master
key.

**Why per-record salt vs a single master salt:**

- An attacker who obtains the database gets N different (ciphertext, salt)
  pairs. Without the master password, they would need to run scrypt once
  **per record** to test a candidate password. A single master salt would
  let them run scrypt once and bulk-decrypt.
- Gives us forward flexibility: rotating a single record re-randomizes its
  salt without invalidating others.
- Cost: two scrypt calls per encrypt/decrypt (one for the master key at
  boot, one per record at use time). The master-key call happens once at
  `new KeyStore()`; only the per-record call is on the hot path.

### Master-KDF salt

The salt fed into the **first** scrypt call (deriving the root key from
`STEWARD_MASTER_PASSWORD`) is:

- `process.env.STEWARD_KDF_SALT` if set (recommended, ≥32 hex chars)
- Otherwise the literal `"steward-vault-v1"` with a `console.warn` in
  production (`keystore.ts` lines ~37–46).

The fallback is a conscious compatibility tradeoff — it means an upgrade does
not silently break existing deployments — but it weakens precomputation
resistance. The warning exists. The threat model calls this out.

**Action item for operators:** set `STEWARD_KDF_SALT` in production. This is
documented in `docs/deployment.md` (and will be re-emphasized in the
forthcoming `SECURITY.md`).

### JWT signing: HS256

Session tokens, agent tokens, and platform-scoped tokens are all signed with
HS256 via `jose` (`packages/auth/src/session.ts` line ~63, plus
`packages/api/src/routes/auth.ts` line ~171 and
`packages/api/src/services/context.ts` lines ~59, ~72).

The signing secret is `STEWARD_SESSION_SECRET` (or, for legacy embedded
mode, derived from `STEWARD_MASTER_PASSWORD`).

**Why HS256 (vs RS256 / ES256):**

- HS256 is symmetric. The issuer and the verifier must share the secret. In
  Steward, **the issuer and the verifier are the same process** (or
  processes that already share `STEWARD_SESSION_SECRET` via env — API and
  proxy, see `docker-compose.yml`). Asymmetric signing buys nothing.
- RS256/ES256 would matter if third-party services needed to verify Steward
  JWTs without calling back to the API. That is not a use case we support
  today, and when it arrives (e.g. for a dashboard SSO federation) we will
  add a second key type, not replace HS256.
- HS256 avoids key-distribution and JWKS-endpoint surface area.

**Tradeoff accepted:** anyone with `STEWARD_SESSION_SECRET` can forge
arbitrary JWTs. This secret is therefore treated with the same care as
`STEWARD_MASTER_PASSWORD` (and is in the same docker-compose env block).
The threat model treats "attacker with `STEWARD_SESSION_SECRET`" as
equivalent to "attacker with code execution on the API" — total compromise.

We pin `algorithms: ["HS256"]` in `jwtVerify()` (session.ts line ~83) to
avoid the classic `alg: "none"` and `alg: "RS256" ↔ "HS256"` confusion
attacks.

### What is explicitly NOT used in v1

- **HSM / cloud KMS (AWS KMS, GCP KMS, Azure Key Vault).** Steward v1 holds
  the master password in the API process. This is a deliberate simplicity
  choice: self-hosted deployments on a single box should Just Work without
  a cloud account. The Roadmap in VISION.md lists "Pluggable key storage
  backends (AWS KMS, Hashicorp Vault)" as a planned follow-up. The
  abstraction point is `KeyStore` — a future `KmsKeyStore` implements the
  same `encrypt` / `decrypt` interface and routes to the KMS.
- **TEE / SGX / Nitro Enclaves.** Out of scope. If a user needs
  enclave-isolated signing, they should deploy a dedicated signer behind
  the vault-as-separate-process design sketched in
  [ADR-0004](./0004-policy-enforcement-model.mdx).
- **Threshold / MPC signing.** Not a goal. Steward's value proposition is
  "sober single-party custody with strong policy," not "trust-minimized
  signing." MPC and threshold-signing platforms are out of scope.

## Consequences

- **Positive:** no native dependencies, no cloud account required, all
  primitives are in `node:crypto` and the behavior is inspectable in one
  ~100-line file.
- **Negative:** the master password is in process memory. Not a unique
  weakness (every software-only wallet has this), but we should be honest
  about it rather than hand-wave.
- **Future flexibility:** the `KeyStore` class is the single point of
  abstraction. A KMS-backed variant, a different KDF, or an AEAD swap are
  all contained changes.
